# 1. Figma 与 CSS 的对应关系

今天主要围绕 **Figma设计稿如何对应前端CSS实现** 进行了理解。

当前项目中的UI确认工作，并不是完整的前端开发，而是根据设计稿检查实际页面是否符合要求。

重点确认项目：

- 文字是否加粗
- 文字大小是否一致
- 行高是否一致
- 页面余白是否符合设计稿

对应关系如下：

| 检查项目 | Figma（日文） | CSS | 浏览器确认 |
|---|---|---|---|
| 字体粗细 | スタイル / W值 | font-weight | Computed |
| 字体大小 | サイズ | font-size | Computed |
| 行高 | 行の高さ / 14/20中的20 | line-height | Computed |
| 内部空白 | パディング | padding | Computed |
| 元素间距离 | 間隔 | gap / margin | Computed |

# 2. Figma 字体标记 W3/W6 的含义

Figma设计系统中经常会看到：

```
W6 30
```

这种标记。

通常：

- W = Weight（字体字重）
- 数字 = 字重等级
- 后面的数字 = 字号

对应 CSS：

| Figma | CSS font-weight | 含义 |
|---|---|---|
| W3 | 300 | Light |
| W4 | 400 | Regular |
| W5 | 500 | Medium |
| W6 | 600 | SemiBold |
| W7 | 700 | Bold |
| W8 | 800 | ExtraBold |
| W9 | 900 | Black |

例如：

```
W6 30
```

表示：

```css
font-weight:600;
font-size:30px;
```

需要注意：

不同设计系统可能使用不同表示方式：

例如：

- Regular
- Medium
- Bold

或者：

- W4
- W5
- W7

本质都是描述字体粗细。

# 3. Figma 中余白与 CSS 的关系

今天确认了一个容易混淆的点：

> Figma中的「余白」不一定等于CSS中的padding。

需要区分：

- Padding
- Gap
- Margin

---

## 3.1 Padding（容器内部空间）

这是Figma最容易识别的信息。

例如按钮：

```
+----------------+
|                |
|     登录       |
|                |
+----------------+
```

Figma Auto Layout：

```
Padding Top:8
Padding Bottom:8
Padding Left:16
Padding Right:16
```

对应：

```css
padding:8px 16px;
```

AI生成代码时，这部分准确率最高。

---

## 3.2 Gap（元素之间距离）

如果Figma使用Auto Layout：

例如：

```
Button

   16px

Button
```

Figma记录：

```
Item Spacing = 16
```

CSS通常：

```css
display:flex;
gap:16px;
```

或者：

```css
display:grid;
gap:16px;
```

---

## 3.3 Margin（通常是AI推断）

需要注意：

> Figma没有Margin属性。

例如：

两个元素之间有24px距离：

Figma知道：

- 元素A的位置
- 元素B的位置
- 两者之间的距离

但是不知道具体CSS实现。

可能生成：

```css
margin-bottom:24px;
```

也可能：

```css
gap:24px;
```

甚至：

```css
padding-bottom:24px;
```

只要最终视觉效果一致。

因此：

做Figma检查时，不应该寻找：

> "这个margin在Figma哪里？"

应该确认：

> "实际页面是否还原了设计稿要求的间距？"

# 4. Figma → Gemini → LWC 的AI开发流程

目前项目中可能采用类似流程：

```
Figma Design
      ↓
AI读取设计信息
      ↓
Gemini生成代码
      ↓
LWC实现
```

AI比较容易获取：

- 尺寸
- 颜色
- 字体
- 圆角
- Padding
- Gap
- Auto Layout结构

但是以下内容仍然需要开发者判断：

- Component拆分方式
- CSS组织方式
- Salesforce标准组件选择
- 数据取得方式
- 可维护性

因此：

AI可以提高开发效率，但不能完全替代开发设计。

# 6. Markdown图片管理

推荐目录：

```

│
├── note.md
│
└── assets
    ├── figma-button.png

    └── difference.png
```

- 文件体积变大
