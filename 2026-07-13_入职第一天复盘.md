## 今天接触到的核心内容

### 1. Salesforce Knowledge 对外暴露方式
Knowledge是内容存储对象，Experience Cloud只是展示渠道之一。对外暴露有4条路径：
- Experience Cloud公开站点（Guest User可访问）
- Experience Cloud登录门户（Partner/Customer）
- REST/GraphQL API直接对接自建前端（Headless模式）
- Agentforce检索Knowledge作为AI问答知识源

**对象模型**：`Knowledge__kav`（文章的具体版本记录，按语言/版本区分）vs `Knowledge__ka`（文章本体）。这是标准知识点，之后配置权限、写SOQL查Knowledge时会用到。

### 2. 项目的实际技术方案（今天讨论得出）
限制条件：不能跳转、全面面向Guest User、尽量不改老系统、不大增开发量。

收敛方案：
```
官网（不跳转，用户无感知）
   └── iframe 嵌入
        └── Experience Cloud（Guest User 可读 Knowledge）
             └── 每篇 Knowledge 有独立公开 URL
                  └── Google 直接抓取该 URL 做 SEO
```
关键逻辑：iframe解决"用户体验上不跳转"，独立URL+Guest User开放解决"SEO仍然生效"——两个目标本来会冲突，用这个组合方案同时满足。

### 5. Markdown 笔记方法（工具/记录习惯，非Salesforce业务内容）

**为什么用 md 记笔记/文档**：
- 纯文本，能被 git 管理、看 diff，Word 做不到
- 一份正本可以导出成多种格式（HTML/Word），不用两边分别维护
- 写法轻量（`#`标题、`-`列表、` ``` `代码块），不用手动排版
- GitHub、VS Code 等到处都能原生渲染，不用装专门软件
- 体积小，适合日报、踩坑记录这种频繁新增的小文件

**md 里插入图片的做法**：
- 基本语法：`![替代文字](相对路径/图片.png)`，路径用相对路径，不要用本地绝对路径（换机器/发布到 GitHub 会裂）
- 图片单独放 `assets/文档名/` 子文件夹，别和 md 混在根目录
- 需要控制大小/居中时，vanilla markdown 语法做不到，要混写 HTML：
  ```html
  <img src="assets/xxx.png" width="500">
  <p align="center"><img src="assets/xxx.png" width="500"></p>
  ```
- 命名要有意义（`guest-user-profile-settings.png`），不要 `image1.png` 这种

**VS Code 快速预览 md**：`Ctrl+Shift+V`（单独面板预览）或 `Ctrl+K V`（右侧分屏，边写边看渲染效果）
